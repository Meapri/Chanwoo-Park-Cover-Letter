EdgeTranslate-v3: 여러 번역 경로를 안정적으로 묶는 브라우저 확장

EdgeTranslate-v3는 페이지 번역을 하나의 API 호출로 끝내지 않고, 브라우저 확장 안에서 여러 provider와 UI 상태를 함께 다루는 프로젝트입니다. Manifest V3의 서비스 워커 수명, 페이지 DOM 변화, 스트리밍 응답, 캐시 상태가 동시에 움직이는 환경에서 번역 경험이 끊기지 않도록 구조를 나눴습니다.

마주한 문제 번역 경로와 UI 상태가 따로 흔들리는 상황

provider별 응답 형식, MV3 수명, 페이지 DOM 변화, 패널 상태가 어긋나면 사용자는 번역이 멈춘 것처럼 느낍니다.

설계 방향 provider 라우팅과 캐시 계층 분리

번역 경로를 한곳에서 선택하고, IndexedDB 캐시와 fallback 흐름을 별도 책임으로 나누어 반복 요청 비용을 줄였습니다.

확인한 결과 결과 패널까지 이어지는 번역 흐름

라우팅, 캐시, content panel이 같은 흐름에서 움직이도록 연결해 결과가 사용자 화면까지 자연스럽게 도달하게 했습니다.

구현 포인트

  • pageTranslate.js에서 Google, AI, DOM, Gemini Nano, Local, Chrome Built-in 번역 경로를 선택하는 흐름을 구성했습니다.
  • translationPersistentCache.js에서 IndexedDB 캐시, URL 해시 프리패치, TTL 정리를 맡겨 반복 번역의 비용을 낮췄습니다.
  • Panel.jsx에서는 streaming result, floating/fixed 모드, docking, drag velocity, TTS를 실제 사용 흐름에 맞춰 연결했습니다.

살펴볼 파일

  • packages/EdgeTranslate/src/background/library/pageTranslate.js
  • packages/EdgeTranslate/src/background/library/translationPersistentCache.js
  • packages/EdgeTranslate/src/content/display/Panel.jsx

왜 중요한가

페이지 번역은 텍스트를 보내고 결과를 받는 기능처럼 보이지만, 실제 제품에서는 DOM 분할, provider 선택, 캐시, UI 반응성이 함께 맞아야 합니다. 이 프로젝트는 그 흐름을 확장 프로그램 안에서 견디도록 만든 작업입니다.

어떻게 읽으면 좋은가

pageTranslate.js에서 번역 경로를 고르고, translator 계층이 외부 또는 로컬 모델과 통신하며, content panel이 결과를 보여주는 순서로 보면 전체 구조가 잡힙니다.

엔지니어링 관점

브라우저 API 제약, 비동기 네트워크, 프론트엔드 인터랙션, 캐시 설계를 한 제품 안에서 함께 다룬 경험입니다. 외부 provider가 흔들려도 사용자 흐름을 유지하는 구조를 고민했습니다.
