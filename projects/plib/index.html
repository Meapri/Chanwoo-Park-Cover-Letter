Plib: Android 앱 내부에서 Linux/glibc 런타임을 다루는 실험

Plib는 Android app-private 공간 안에서 root 권한 없이 Linux/glibc arm64 런타임을 실행해 보는 실험 프로젝트입니다. Android 샌드박스, rootfs 설치, native binary 실행, JNI 진단, EGL/GLES bridge가 만나는 지점을 문서와 코드로 나누어 확인했습니다.

마주한 문제 권한 상승 없이 Linux/glibc 환경을 실행하는 제약

app-private rootfs, native binary 실행, 동적 라이브러리 로드, 그래픽스 출력이 Android 보안 모델 안에서 함께 맞아야 합니다.

설계 방향 clean-room 문서와 ALR runtime 계층 분리

요구사항과 구현 경계를 문서화하고, config, path, env, exec, hook, interposer, trampoline으로 실행 계획을 나눴습니다.

확인한 결과 설치부터 GPU bridge까지 이어지는 진단

rootfs safe extraction, runtime launch plan, JNI report, EGL/GLES probe, Android Surface rendering을 한 흐름에서 확인합니다.

구현 포인트

  • runtime_report.cpp에서 JNI report, EGL/GLES pbuffer, Android Surface rendering, dlopen probe를 분리했습니다.
  • alr_runtime의 config, path, env, exec, hook, interposer, trampoline 계층으로 실행 계획을 나눴습니다.
  • Kotlin의 RootfsInstaller, RootfsInstallPlan, NativeCommandRunner로 rootfs 안전 추출과 실행 흐름을 관리했습니다.

살펴볼 파일

  • app/src/main/cpp/runtime_report.cpp
  • app/src/main/cpp/alr_runtime
  • app/src/main/java/dev/chanwoo/androlinux

왜 중요한가

Android 앱 안에서 Linux/glibc 런타임을 다루려면 OS 샌드박스, 파일 시스템 경로, 동적 링킹, native 실행, 그래픽스 출력이 모두 연결되어야 합니다. Plib는 그 경계를 작게 나눠 어디서 깨지는지 확인하는 실험입니다.

어떻게 읽으면 좋은가

먼저 docs/product-requirements.md와 clean-room-protocol.md에서 제약을 확인하고, Kotlin 설치 흐름에서 rootfs가 준비되는 과정을 보면 좋습니다. 이후 alr_runtime과 runtime_report.cpp를 따라가면 native 실행과 GPU bridge가 이어집니다.

엔지니어링 관점

앱 레벨 UI를 넘어 Android NDK, C++ 런타임, Linux 실행 모델, 그래픽스 bridge를 함께 읽은 경험입니다. 제약이 강한 환경에서 문제를 쪼개고 검증 가능한 경계로 만드는 방식이 드러납니다.
