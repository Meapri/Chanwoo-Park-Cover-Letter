mlx-swift-lm-gemma4-mtp: 최신 로컬 LLM 실험을 위한 MLX Swift 하위 계층 확장

mlx-swift-lm-gemma4-mtp는 MLX Swift 기반 로컬 LLM 실험을 위해 모델 등록과 draft 모델 경로를 확장한 프로젝트입니다. 단순히 upstream을 보관한 저장소가 아니라, Gemma4 MTP assistant와 DFlash draft model, Qwen3.5 계열 로딩 경로를 Swift 코드로 연결했습니다.

마주한 문제 새 모델 구조가 Swift 런타임에 바로 올라오지 않는 상황

config 구조, model type, KV cache, tool-call 형식이 모델마다 달라 upstream 기본 경로만으로는 실험하기 어렵습니다.

설계 방향 모델 등록과 draft path를 명시적으로 확장

LLMModelFactory에 새 model type을 등록하고, target hidden과 shared KV를 draft model로 넘기는 경로를 만들었습니다.

확인한 결과 서버 프로젝트가 의존할 수 있는 Swift 패키지

Gemma4 Swift MTP Server가 이 패키지를 의존성으로 삼아 실제 server-side generation path를 구성합니다.

구현 포인트

  • LLMModelFactory.swift에 gemma4, gemma4_assistant, dflash_draft, qwen3_5 계열을 등록했습니다.
  • Gemma4Assistant.swift는 target model의 embedding과 shared KV state를 받아 draft token block을 생성합니다.
  • DFlashDraft.swift는 mask token, target layer id, target hidden conditioning, cache update를 Swift 모델로 옮겼습니다.
  • Tool/Parsers 계층에는 Gemma와 Qwen3.5 계열의 tool-call 형식 처리가 추가되어 agent 호환성을 넓힙니다.

왜 중요한가

로컬 LLM 실험은 표면 API보다 하위 모델 계층에서 자주 막힙니다. 새 아키텍처를 Swift 런타임에 맞게 읽고 연결해야 서버와 앱이 그 위에 설 수 있습니다.

어떻게 읽으면 좋은가

모델 factory 등록에서 시작해 Gemma4 target model, assistant drafter, DFlash draft model 순서로 따라가면 MTP 서버가 왜 이 확장 패키지를 필요로 하는지 보입니다.

엔지니어링 관점

upstream을 단순히 가져오는 데서 멈추지 않고, 새 모델의 구조와 Swift 타입 시스템 사이를 맞춘 작업입니다. 모델 내부 구현을 제품 API와 연결하는 기반 경험입니다.
