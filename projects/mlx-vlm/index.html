MLX/VLM Experiments: 공개 fork에서 런타임 호환성의 경계를 읽은 실험

이 작업은 MLX-Swift, mlx-swift-lm, mlx-vlm 공개 fork를 바탕으로 진행한 분석과 확장 실험입니다. 원저작 프로젝트의 성과와 경계를 분명히 두고, 모델 로드 계획, 멀티모달 입력, KV cache, OpenAI/Ollama 호환 API가 어떻게 이어지는지 직접 읽어 본 과정입니다.

마주한 문제 로컬 AI 런타임과 외부 API 규격의 간격

모델 로드, tokenizer, 멀티모달 입력, KV cache, streaming 응답이 각각 다른 계층에서 맞물립니다.

설계 방향 원본 구조를 존중하는 호환성 계층 실험

공개 fork의 기존 런타임 흐름을 따라가며 OpenAI/Ollama 호환 타입과 SSE streaming 경로를 실험했습니다.

확인한 결과 VLM 입력부터 응답 streaming까지의 연결

QwenVL prompt/image planning, embedding merge, generation plan, SSE streaming이 이어지는 지점을 확인했습니다.

구현 포인트

  • MLXVLMCore의 ModelLoadPlanner, GenerationRuntimePlan, GenerationSamplingPlan을 따라 추론 준비 과정을 읽었습니다.
  • QwenVLPromptBuilder, QwenVLImageInputPlanner, QwenVLEmbeddingMerge 경로에서 멀티모달 입력이 합쳐지는 방식을 확인했습니다.
  • CLI에서 tokenize/detokenize, OpenAI chat/completions/responses, Ollama generate/chat/embed, SSE streaming과 KV cache 경로를 실험했습니다.

살펴볼 파일

왜 중요한가

로컬 AI 런타임은 모델을 실행하는 것만으로 끝나지 않습니다. 외부 앱이 기대하는 API 형태, 멀티모달 입력 전처리, streaming 응답, cache 상태가 함께 맞아야 실제 도구로 쓸 수 있습니다.

어떻게 읽으면 좋은가

MLXVLMCore에서 모델 로드와 generation plan을 먼저 따라가고, QwenVL 계층에서 이미지 입력이 프롬프트와 임베딩으로 합쳐지는 흐름을 보면 좋습니다. 이후 CLI와 호환 API 타입을 보면 서버 경계가 드러납니다.

엔지니어링 관점

낯선 오픈소스 코드베이스를 빠르게 읽고, 기존 구조를 존중하면서 필요한 호환성 실험을 진행한 경험입니다. AI 런타임 내부와 제품 API 경계를 함께 이해하는 역량을 보여줍니다.
